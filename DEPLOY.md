# Деплой на Ukrline (ISPmanager, shared Node.js хостинг)

## Шаг 1. Создай Node.js-сайт в ISPmanager

1. Войди в ISPmanager → **Сайты** (WWW-домены) → **Создать сайт**.
2. Укажи свой домен (или поддомен, например `game.твой-домен.ua`).
3. В типе обработчика выбери **Node.js**.
4. Версия Node.js: выбери самую свежую из доступных (минимум 18, лучше 20/22+).
5. **Стартовый файл приложения**: `server/index.js`.
6. Сохрани. Панель создаст каталог сайта (обычно `~/www/<домен>/`).

## ⚠️ Чтобы игры не пропадали при передеплое (важно!)

База `sea-battle.db` хранит все активные игры, лидерборд и игроков. Если она лежит
**внутри папки сайта**, то при деплое нового коммита (git-деплой или «чистая»
распаковка архива перетирает рабочую папку целиком) база сбрасывается — тогда после
деплоя игры и лидерборд «обнуляются».

**Теперь по умолчанию база создаётся ВНЕ папки сайта** — в домашней директории
`~/.sea-battle/sea-battle.db`, которую деплой не трогает. В большинстве случаев менять
ничего не нужно: при старте в логе будет видно
`🗄  База данных: /home/<логин>/.sea-battle/sea-battle.db`.

Если же в логе путь указывает **внутрь папки сайта** (появится предупреждение
`⚠️ База лежит ВНУТРИ папки сайта`) — задай переменную окружения `DB_PATH` вручную:

1. Создай вне каталога сайта папку для данных, например `~/sea-battle-data/`.
2. В ISPmanager на странице Node.js-сайта добавь переменную окружения:
   `DB_PATH=/home/<твой-логин>/sea-battle-data/sea-battle.db`
   (точный путь до домашней папки уточни в панели).
3. Перезапусти приложение. В логе при старте появится `🗄  База данных: <путь>` и
   `игр в базе: N` — так видно, что база на месте и переживёт следующий деплой.

⚠️ **Важно при первом переходе на новую версию:** база переезжает на новое место,
поэтому игры/лидерборд, накопленные в старом файле внутри папки сайта, не подхватятся
автоматически. Если хочешь их сохранить — один раз скопируй старый `sea-battle.db`
(из каталога сайта) в `~/.sea-battle/` через файл-менеджер до запуска. Дальше всё
переживает деплой само.

Дополнительно код после каждого сохранения «сливает» данные в основной файл базы,
так что даже файлы-спутники `-wal`/`-shm` больше не критичны.

## Шаг 2. Загрузи код

1. Возьми архив `sea-battle-deploy.zip` (он уже собран, лежит в корне проекта).
2. ISPmanager → **Менеджер файлов** → зайди в каталог сайта → **Загрузить** архив → **Извлечь** его прямо туда.
3. Проверь, что в корне сайта лежат `package.json`, `server/`, `public/`.

## Шаг 3. Установи зависимости

- В ISPmanager на странице сайта (раздел Node.js) есть кнопка установки npm-зависимостей —
  нажми её (выполнится `npm install`).
- Зависимостей три: `express`, `socket.io`, `better-sqlite3`. Последняя — нативная,
  но при установке она скачивает готовый бинарник под Linux, компиляция обычно не нужна.

## Шаг 4. Запусти и проверь

1. На странице сайта нажми **Перезапустить** приложение.
2. Открой `https://твой-домен/` — должна появиться главная «Морской бой на листке в клетку».
3. Создай баттл, открой ссылку в режиме инкогнито вторым игроком, начни игру и сделай пару ходов.
4. Логи приложения смотри в панели (раздел Node.js → журнал) — туда пишется
   `⚓ Sea Battle: http://localhost:<порт>` при старте.

## Что уже учтено в коде

- Порт берётся из `process.env.PORT` — ISPmanager подставляет свой, ничего настраивать не надо.
- База SQLite создаётся сама при первом старте (файл `sea-battle.db` рядом с `package.json`).
- Socket.IO при проблемах с WebSocket-проксированием сам откатывается на long-polling —
  игра останется рабочей, просто чуть медленнее доставка ходов.

## Если что-то пошло не так

| Симптом | Что делать |
|---|---|
| `npm install` падает на `better-sqlite3` | Напиши мне — заменю на встроенный `node:sqlite` (нужен Node 22+) за пару минут. |
| Сайт отвечает 502/503 | Проверь стартовый файл (`server/index.js`) и версию Node ≥ 18; смотри журнал в панели. |
| Игра открывается, но ходы не приходят | Открой консоль браузера (F12): если Socket.IO не коннектится даже polling'ом — пришли мне текст ошибки. |
| Приложение «засыпает» | В FAQ хостера заявлен автоматический мониторинг и рестарт процессов; состояние игр в SQLite, так что партии переживают рестарты. |

## Вход через Google (опционально)

1. Зайди на https://console.cloud.google.com/ → создай проект (любое имя).
2. **APIs & Services → OAuth consent screen**: тип External, заполни название
   приложения и свою почту, добавь себя в Test users (или опубликуй приложение —
   для простого Sign-In верификация не требуется).
3. **APIs & Services → Credentials → Create Credentials → OAuth client ID**:
   - Application type: **Web application**;
   - **Authorized JavaScript origins**: `https://твой-домен` (точный адрес сайта, без слэша в конце);
   - Redirect URI не нужен.
4. Скопируй **Client ID** (вида `1234-abc.apps.googleusercontent.com`).
5. В ISPmanager на странице Node.js-сайта добавь переменную окружения
   `GOOGLE_CLIENT_ID=<этот id>` и перезапусти приложение.

На главной и в окне входа появится кнопка Google. Ник по-прежнему вводится свой.

## Письма «Поторопить» (опционально)

1. В ISPmanager создай почтовый ящик на своём домене, например `game@твой-домен`.
2. На странице Node.js-сайта добавь переменные окружения:
   - `SMTP_HOST` — почтовый сервер хостинга (обычно `mail.твой-домен`, см. настройки ящика в ISPmanager);
   - `SMTP_PORT` — `465` (SSL) или `587`;
   - `SMTP_USER` — `game@твой-домен`;
   - `SMTP_PASS` — пароль ящика;
   - `SMTP_FROM` — `Морской бой <game@твой-домен>`;
   - `BASE_URL` — `https://твой-домен` (для ссылки «Сделать ход» в письме).
3. Перезапусти приложение. В логе при старте появится «📯 Почтовые уведомления включены».

Письмо получает AFK-игрок, если он входил через Google (иначе у нас нет его почты —
тогда «Поторопить» просто включает 10-минутный таймер).

## Обновление версии игры

Просто заливаешь новый архив поверх (или отдельные изменённые файлы) через менеджер
файлов и жмёшь «Перезапустить». База `sea-battle.db` не входит в архив — прогресс и
лидерборд сохранятся.
